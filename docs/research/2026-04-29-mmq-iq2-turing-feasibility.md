# Research Report: MMQ for IQ2_XXS on Turing (sm_75) — Feasibility for tg128 +20%

**Date:** 2026-04-29
**Author:** cwe-researcher
**Repo:** `/mnt/d/repos/llama-tq` (llama-tq fork at master)
**Target:** Qwen3.6-35B-A3B-IQ2_XXS, 2× RTX 2060, tg128 ~83 t/s → ≥ 100 t/s

---

## Executive Summary

The mission as briefed assumes that **MMQ does not exist for IQ2_XXS on sm_75** and that adding it would unlock a +20% TG win on MoE tg128. Audit of the current tree contradicts that assumption on three independent points:

1. **MMQ for IQ2_XXS already exists in this fork**, fully wired through dispatch, template-instance file, load-tiles, and DP4A + Turing-MMA vec-dot kernels. No new implementation is required.
2. **`ggml_cuda_should_use_mmq` returns `true` for IQ2_XXS on Turing** when `ne11 < 64`, so MMQ is the *eligible* path on paper.
3. **However, on `mul_mat_id` (the path used by Qwen3.6 MoE during tg), MMVQ is preferred over MMQ for any batch size ≤ 8.** This is enforced at `ggml-cuda.cu:2449-2462` via `get_mmvq_mmid_max_batch`, which returns the default `MMVQ_MAX_BATCH_SIZE=8` for IQ2_XXS on Turing+ (`mmvq.cu:130-140`). For tg128, `ne2=1` always, so MMVQ wins unconditionally.

The right framing of the question is therefore not "implement MMQ" but **"force MMQ instead of MMVQ for IQ2_XXS on Turing in MoE tg, and measure"**. This is a one-line change (or env var + 5-line conditional) and is empirically already gated against by upstream maintainers — see PR #20905 reasoning below — meaning past benchmarks on other arches found MMVQ ≥ MMQ for IQ2_XXS at small batch on NVIDIA. Whether that result holds on Turing TU106 with this fork's grid-staging shmem fix (commit `36743ad76`) is **unverified** and is the only remaining experiment that can answer feasibility.

**Bottom line preview:** No-Go on the mission as stated (there is nothing to implement). Recommend instead a 1-day A/B benchmark forcing MMQ via `GGML_CUDA_FORCE_MMQ=1` plus a per-token small-MMQ kernel tweak. Realistic upside if MMQ wins is **+5 to +12%**, not +20%. The path to +20% on this hardware lives elsewhere (layer split, occupancy, asymmetric K/V — all already explored in fork memory).

## 1. MMQ Landscape Audit on sm_75

### 1.1 MMQ supports IQ2_XXS in this fork

`ggml/src/ggml-cuda/mmq.cu:44-46` — MMQ dispatch entry exists:

```cpp
case GGML_TYPE_IQ2_XXS:
    mul_mat_q_case<GGML_TYPE_IQ2_XXS>(ctx, args, stream);
    break;
```

`ggml/src/ggml-cuda/template-instances/mmq-instance-iq2_xxs.cu:1-5` — template instance file present:

```cpp
// This file has been autogenerated by generate_cu_files.py, do not edit manually.
#include "../mmq.cuh"
DECL_MMQ_CASE(GGML_TYPE_IQ2_XXS);
```

`ggml/src/ggml-cuda/mmq.cuh:200` — DP4A tile-size mapping: IQ2_XXS uses `MMQ_DP4A_TXS_Q8_0`, meaning after dequant it stores int8 quants identically to Q8_0.

`ggml/src/ggml-cuda/mmq.cuh:245` — Turing/AMD MMA tile mapping: IQ2_XXS → `MMQ_MMA_TILE_X_K_Q8_0`.

`ggml/src/ggml-cuda/mmq.cuh:81-86` — q8_1 layout for activations: `MMQ_Q8_1_DS_LAYOUT_D4` (one fp32 scale per 32 values).

`ggml/src/ggml-cuda/mmq.cuh:2766-2826` — `load_tiles_iq2_xxs<mmq_y, need_check>`: walks each block, reads 32 bits of quants per call, looks up `iq2xxs_grid[aux8[l]]` (returning `uint2` packed bytes), applies sign mask via `__vsub4(grid_pos.x ^ signs0, signs0)`, and writes int8 packed quads into the shared-memory tile at offset `MMQ_MMA_TILE_X_K_Q8_0` for the MMA path or `(2*MMQ_TILE_NE_K + 1)` for DP4A. Per-block scale `d * ls / 8` (line 2821) is stored in `x_df` parallel to q8_0 layout.

After this, the standard `vec_dot_q8_0_q8_1_*` family of dot kernels is reused (no IQ2-specific dot).

### 1.2 mmq_supported gating on Turing

`ggml/src/ggml-cuda/mmq.cu:285` — IQ2_XXS in the `mmq_supported = true` switch.
`ggml/src/ggml-cuda/mmq.cu:304-318` — gate logic:

```cpp
if (turing_mma_available(cc)) { return true; }                          // sm_75+
if (ggml_cuda_highest_compiled_arch(cc) < GGML_CUDA_CC_DP4A) { return false; }
#ifdef GGML_CUDA_FORCE_MMQ
return true;
#endif
if (GGML_CUDA_CC_IS_NVIDIA(cc)) {
    return !fp16_mma_hardware_available(cc) || ne11 < MMQ_DP4A_MAX_BATCH_SIZE;  // 64
}
```

For sm_75 (Turing, has fp16 mma): `mmq_supported = true` if `ne11 < 64`. For tg128, `ne11=1`, so MMQ is eligible.

### 1.3 So why isn't MMQ being used today?

Because the mul_mat_id dispatcher at `ggml/src/ggml-cuda/ggml-cuda.cu:2447-2467` checks **MMVQ first**:

```cpp
if (ne2 <= MMVQ_MAX_BATCH_SIZE) {                              // 8
    if (ggml_is_quantized(src0->type) && !is_tq_weight_type) { // IQ2_XXS qualifies
        const int mmvq_mmid_max = get_mmvq_mmid_max_batch(src0->type, cc);
        if (ne2 <= mmvq_mmid_max) {
            ggml_cuda_mul_mat_vec_q(ctx, src0, src1, ids, dst);  // ← TG WINS HERE
            return;
        }
    }
    ...
}
if (ggml_cuda_should_use_mmq(...)) {                            // never reached for tg
    ggml_cuda_mul_mat_q(ctx, src0, src1, ids, dst);
    return;
}
```

`ggml/src/ggml-cuda/mmvq.cu:130-140` — `get_mmvq_mmid_max_batch_turing_plus`:

```cpp
case GGML_TYPE_IQ2_S:   return 7;
case GGML_TYPE_IQ3_S:   return 6;
case GGML_TYPE_IQ3_XXS: return 7;
case GGML_TYPE_MXFP4:   return 7;
case GGML_TYPE_Q2_K:    return 7;
case GGML_TYPE_Q3_K:    return 5;
default:                return MMVQ_MAX_BATCH_SIZE;             // = 8 → IQ2_XXS hits this
```

IQ2_XXS is **not** explicitly listed → falls through to default `MMVQ_MAX_BATCH_SIZE=8`. For any tg call (`ne2=1`), MMVQ runs. MMQ is dead code on this path.

### 1.4 Upstream provenance

The per-type per-arch tuning came from PR #20905 ("MOE GEMV optimization for batch>1") which cherry-picked from PR #20885. From [github.com/ggml-org/llama.cpp/pull/20905](https://github.com/ggml-org/llama.cpp/pull/20905):

> "small_k optimization only for cases where it benefits ... on NVIDIA Turing or newer disable for iq3_xxs and iq3_s."
> `MMVQ_MMID_MAX_BATCH_SIZE` raised 4 → 8.

Implication: maintainers benchmarked small-batch MMVQ vs MMQ across types and arches. Types absent from `get_mmvq_mmid_max_batch_turing_plus` (including IQ2_XXS) implicitly mean **MMVQ at full batch (≤8) was at least tied** — otherwise they would have lowered the cap.

This is not absolute proof MMQ loses on this fork's hardware. The grid-staging fix (commit `36743ad76` + merge `d84a1598a`, +11% on mmvq) post-dates upstream's benchmarks and altered mmvq's cost model (eliminated divergent gather into `__device__ const`). Worth a re-bench, but starting prior is "MMVQ ≥ MMQ for IQ2_XXS on Turing TG."

## 2. DP4A Throughput Math on Turing (TU106)

RTX 2060 = TU106 = sm_75, 30 SMs, base ~1365 MHz, boost ~1680 MHz.

DP4A is `__dp4a(int a, int b, int c)`: 4× int8 multiply-accumulate per cycle, returning int32. On Turing it issues on the FP32 SP datapath (no separate INT IMMA pipe at sm_75; that's Ampere's INT8 tensor cores).

**Per-SM int8 ops/cycle:** 64 FP32-equivalent lanes × 4 ops (MAC = 2 ops, but DP4A does 4 multiply-adds packed) = 256 int8 ops/cycle. Some sources count DP4A as 8 ops (4 mul + 4 add). Using NVIDIA's "8 ops per DP4A" convention:

- 30 SMs × 64 lanes × 8 ops × 1.68 GHz ≈ **25.8 TOPS int8** (theoretical peak DP4A).

That is roughly 2× the FP16 fma peak on TU106 (~12.9 TFLOPS via tensor cores, ~6.4 TFLOPS via SIMT FP16x2). So **DP4A *is* faster than fp16 fma on Turing** — but only by ~2×, not the 4× seen on Ampere where tensor cores have integer mode.

Note: NVIDIA Turing whitepaper, [Turing Architecture Whitepaper](https://images.nvidia.com/aem-dam/Solutions/design-visualization/technologies/turing-architecture/NVIDIA-Turing-Architecture-Whitepaper.pdf) p. 14, lists "Tensor Cores: 65 TOPS INT8" for TU102. Scaling to TU106 (30 SMs / 72 SMs × 65) ≈ 27 TOPS INT8 via tensor cores — but those are not used by DP4A.

### 2.1 Where the time actually goes in tg128

For MoE token-gen on IQ2_XXS, the bandwidth cost dominates: 2.06 bpw × ~3 B active params × 1 token × 1.0 sample = ~775 MB/token to read. RTX 2060 effective bandwidth ~336 GB/s × 2 GPUs = 672 GB/s aggregate, but with PCIe x16/x4 asymmetry (per `project_gpu00_hardware.md`) and tensor-parallel sync, useful BW is closer to 400-450 GB/s. That puts ceiling at ~580-650 t/s for the GEMV portion alone. Measured ~83 t/s is therefore **far from the BW ceiling**, meaning **compute or scheduling is the bottleneck, not DRAM BW**.

Where mmvq compute dominates:
- Dequant inner loop: `iq2xxs_grid[idx]` is read from `__device__ const` arrays — historically caused divergent global gathers. The recent fix (commit `36743ad76`) staged the grid into shared memory, +11%.
- Sign expansion `__vsub4(grid_pos.x ^ signs0, signs0)`.
- Scale broadcast and float fma accumulator.

**If the dequant is ~20% of mmvq cost (typical for i-quants), MMQ saves it by reusing the dequantized tile across the entire mmq_x dimension. But mmq_x for tg is 1, so that "reuse" buys nothing per-token — the dequant happens once either way.** The only structural win MMQ offers over MMVQ at `mmq_x=1` is:

1. Better tile-load coalescing (q8_1 activations preloaded once into shmem).
2. DP4A vs fp16 fma compute (1.5-2× peak on Turing).
3. Fewer register pressure points if mmq_y is large (>= 64).

These are real but small at batch=1. Estimated upside: **+5 to +12%** if MMQ at mmq_x=1 even wins (it may not — MMVQ kernels are hand-tuned for the exact tile shape and register layout).

**Cannot estimate more precisely without `nsys` trace.** No `docs/perf/` profile data found in the fork.

## 3. IQ2_XXS Structural Compatibility with Int8 Quant-Block Math

`ggml/src/ggml-common.h:595-599`:

```c
typedef struct {
    ggml_half d;          // fp16 super-block scale
    uint16_t  qs[QK_K/8]; // 32 uint16, packed: 8-bit grid index + 7-bit signs
} block_iq2_xxs;          // 66 bytes per 256 vals = 2.0625 bpw
```

Layout: each 256-value super-block has 1 fp16 scale `d`, then 32 uint16 split into 8 groups of 4. Each group encodes:
- 4 × 8-bit grid indices into `iq2xxs_grid[256]` (each entry: `uint2` = 8 packed signed-magnitude 4-bit-ish ternary values).
- 4 × 7-bit sign masks.
- 5-bit per-group multiplicative scale `ls = (aux32 >> 27) | 1` (odd integer 1..31).

After grid lookup + sign apply (load_tiles_iq2_xxs:2799-2807), each 8-bit group expands to 8 int8 values in `[-13..13]` range (grid entries are bounded ternary representations × small ints). **Int8 representation is lossless** — this is exactly what `load_tiles_iq2_xxs` already does today.

The combined per-tile scale `d * ls / 8` is fp32 and pairs with the q8_1 activation scale. This is the same structure as Q8_0 quantized matmul once dequantized, which is why the existing code reuses `MMQ_MMA_TILE_X_K_Q8_0` and the q8_0 vec-dot routines.

**Conclusion:** structural compatibility is a non-issue. The math is settled and already shipped.

## 4. Q8_1 Activation Side

`mmq.cuh:28-47` defines `block_q8_1_mmq` (128 int8 values + 4 fp32 or fp16 scales/sums per 128 values). Activations are quantized once per token via `quantize_mmq_q8_1_cuda` (`mmq.cu:139`) before dispatch.

For IQ2_XXS × q8_1, the inner dot is:
```
sum = Σ_k (iq2_dequantized_int8[k] × q8_1_int8[k])  // 4 packed via DP4A
output = sum × x_scale × y_scale
```

DP4A: 1 cycle for 4 int8×int8 → int32 accumulate. Per K=256 super-block: 64 DP4A ops + tile loads + scale fma. With 30 SMs at 1.68 GHz, theoretical 30×4 warps × 32 lanes × DP4A/cycle = ~6.4 G DP4A/s = ~25 G int8 muls/s. Per 256-value block that's ~0.001 ns of pure compute — clearly NOT compute-bound on Turing for this workload, even at mmq_x=1. **The bottleneck is shared-memory BW, register pressure, and instruction issue overlap, not raw DP4A throughput.**

## 5. Implementation Outline (if forcing MMQ)

Since MMQ already exists, "implementation" reduces to **changing dispatch** so MoE tg uses MMQ instead of MMVQ:

### Option A: Hard fork — add IQ2_XXS to turing_plus skip list

`ggml/src/ggml-cuda/mmvq.cu:130-140`:
```cpp
case GGML_TYPE_IQ2_XXS: return 0;  // force MMQ on Turing+ for IQ2_XXS at all batch sizes
```
This routes ne2=1 mul_mat_id away from mmvq. Combined with `should_use_mmq` returning true at ne11<64, MMQ kicks in.

LOC: ~3 lines. Risk: regresses dense (non-MoE) paths if the same function is consulted there. It is not — mmvq.cu function is specifically `_mmid_` (line 2451 only).

### Option B: Env var gate

```cpp
static bool force_mmq_iq2 = getenv("GGML_CUDA_FORCE_MMQ_IQ2") != nullptr;
if (force_mmq_iq2 && type == GGML_TYPE_IQ2_XXS) return 0;
```
LOC: ~5. Allows runtime A/B without rebuild.

### Option C: Tile-shape autotune at kernel launch

Not viable inside one session — needs benchmark harness changes.

### Files unchanged

- `mmq.cuh` — already has full kernel.
- `template-instances/mmq-instance-iq2_xxs.cu` — already exists.
- `mmq.cu` dispatch — already routes IQ2_XXS.

### Risk assessment

- **PPL drift**: zero. Both paths use identical dequant arithmetic; difference is only how the int8 results are packed and dotted.
- **Kernel-launch overhead**: MMQ kernel is heavier than mmvq for mmq_x=1 (designed for mmq_x=64 or 128). May regress.
- **Register pressure**: MMQ kernels use 64-128 registers per thread on Turing. Turing has 64 KB regfile per SM ÷ 256 threads/CTA ÷ 4 warps = 64 regs/thread budget. MMQ may spill on Turing → killing the +20% goal.
- **Stream-K**: enabled at line 118 (`use_stream_k = cc>=Volta || CDNA`) — on by default for Turing. Stream-K helps small-tile MMQ but adds fixup kernel overhead.

## 7. Recommendation

### Verdict: **No-Go on the mission as briefed. Conditional Go on a 1-day A/B experiment.**

**Why no-go on the original framing:**
1. There is nothing to "implement." MMQ for IQ2_XXS is already in the tree and structurally complete.
2. The +20% goal is unsupported by Turing DP4A math at batch=1. Realistic upside if MMQ wins is +5-12%, and there is upstream-maintainer evidence (PR #20905) suggesting MMVQ ≥ MMQ for IQ2_XXS at small batch on NVIDIA.
3. The bottleneck on this hardware/fork is not where the brief assumed. Per `project_session_2026_04_23_reality_check.md`, mmvq itself is ~28% of frame time, dequant is a sub-fraction of that, and remaining time is in scheduling/sync.

**Conditional Go (1 day):**

Do an A/B benchmark first:
1. Patch `mmvq.cu:130` (Option B above) — env var gate, ~5 LOC.
2. Build with `-DGGML_CUDA_DMMV_X=128 -DGGML_CUDA_MMV_Y=2` or default.
3. Bench `llama-bench -m qwen3.6-35b-a3b-iq2_xxs.gguf -p 0 -n 128 -r 5` with and without `GGML_CUDA_FORCE_MMQ_IQ2=1`.
4. If MMQ wins ≥ 5%, harden via Option A (always-on). If <2% or regresses, revert and document.

Effort: 0.5-1 day. Risk: low (gated, no PPL impact).

**For the +20% target, this is the wrong layer to attack.** Higher-EV paths:
- Re-profile with `nsys` to attribute the remaining 70%+ of frame time. No nsys data exists in the tree today.
- Reduce mul_mat_id launch overhead (the per-expert slice loop at `ggml-cuda.cu:2538-2583` calls `ggml_cuda_mul_mat` 8 times per token — kernel launches dominate when each expert touches only 1 token).
- Tensor-parallel comm: PCIe x4 on the second GPU per `reference_gpu00_hardware.md` is a known asymmetry — quantify all-reduce cost.

---

## Bottom Line

**The MMQ-for-IQ2_XXS path doesn't need building — it's already in the tree and gated off in favor of MMVQ on Turing because upstream maintainers found MMVQ wins at batch ≤ 8.** A 1-day A/B with a 5-line env-var gate is worth running to verify whether the fork's recent grid-staging mmvq fix changes that ordering, with realistic upside +5-12% if MMQ wins. **The +20% target requires attacking kernel-launch overhead in mul_mat_id and PCIe x4 asymmetry, not the mmq/mmvq choice — recommend re-profiling with `nsys` before any further code work.**
