## 3. Template-Änderungen in `fattn-vec-vtq2.cuh`

### 3.1 Relax D-assertion (line 321)

```cpp
static_assert(D == 128, "Phase 3A1 supports D=128 only");
```
Replace with:
```cpp
static_assert(D == 64 || D == 128 || D == 256 || D == 512, "D must be 64/128/256/512");
static_assert(D % QK_VTQ_TRELLIS == 0 || QK_VTQ_TRELLIS % D == 0, "D/128 block ratio must be integral");
```

### 3.2 Multi-block per V-row for D > 128

Currently line 193-194 comment: *"For D=128 each V-row is exactly one VTQ block"*. For D=256 jede row = 2 blocks, D=512 = 4 blocks. Shmem layout must scale:

```cpp
constexpr int blocks_per_row = D / QK_VTQ_TRELLIS;  // 1/2/4 (or handle D<128 branch)
__shared__ half smem_V_cache[nwarps][D < QK_VTQ_TRELLIS ? QK_VTQ_TRELLIS : D];
```

For D=64: one VTQ block covers 2 V-rows. Decode once, slot at `[warp][0..63]` per row — cache reuse across 2 rows.

### 3.3 Warm-loop adjustment (lines 401-416)

Wrap the `vtq2_block_warm` call in a `#pragma unroll` over `blocks_per_row`:

```cpp
#pragma unroll
for (int ib = 0; ib < blocks_per_row; ++ib) {
    const block_t * x_block = (const block_t *) (V + k*nb21) + ib;
    vtq2_block_warm<block_t, K_bits>(x_block, smem_V_cache[threadIdx.y] + ib*QK_VTQ_TRELLIS);
}
__syncwarp();
```

### 3.4 K-dequant — NO changes required

K-side nutzt `vec_dot_KQ = get_vec_dot_KQ<type_K, D, nthreads_KQ>()` (line 154) — bereits fully parametric. F16, Q8_0, KTQ2_1, KTQ3_1 dispatches resolve at template instantiation. No new code needed.

### 3.5 ncols=2 support — REMOVE the abort at line 644

```cpp
GGML_ABORT("flash_attn_ext_vec_vtq2_cached: ncols>=2 not supported in Phase 3A1");
```
Replace with actual `cols_per_block = 2` branch, mirroring the ncols=1 pattern at lines 629-636. Der Kernel body (ncols template param, VKQ[ncols][...] arrays, KQ_reg[ncols], KQ_max[ncols] etc.) is **already** ncols-generic — only the launcher branch is missing.

### 3.6 `nthreads` — KEEP at 128

E11 Triton's `num_warps=1` → 32 threads is the *Triton* sweet spot, not CUDA. Turing reg-file math (spec 3A1 §4): `4 blocks × 128 threads × 128 regs = 65536` already maxes SM. Cutting zu 32 threads loses KQ parallelism. 128 stays.

## 4. cicc Budget

Phase 3A1 ships 2 instances (softcap on/off). Nvcc overhead auf phase2 HEAD war ~+3 min auf `fattn-vec-dispatch-vtq2.cu`.

**3A2 P0 target:** 4 K×V combos × D=128 × ncols∈{1,2} × softcap∈{0,1} = **16 instances** (8 added over 3A1).
Expected cicc delta: **~24 min** extra on the single .cu.

**Gate: > 15 min extra → split.** Well above threshold → **MUST split**.

### File-split strategy

Create three new dispatch .cu files, each including `fattn-vec-vtq2.cuh`:

```
fattn-vec-dispatch-vtq2.cu          (existing, keeps F16/Q8_0 legacy cases)
fattn-vec-dispatch-vtq2-ktq2.cu     (NEW: KTQ2_1 × {VTQ2_2, VTQ3_2, VTQ4_2})
fattn-vec-dispatch-vtq2-ktq3.cu     (NEW: KTQ3_1 × VTQ3_2)
fattn-vec-dispatch-vtq2-extra.cu    (NEW: reserved for D>128 / P1)
```

Each file: ≤ 8 cached instances → ≤ 8 min cicc each → parallel build ~10 min wall clock (vs 24 min serial).

## 5. Register-Budget per D-Wert

Phase 3A1 `__launch_bounds__(128, 4)` at fattn-vec-vtq2.cuh:93. For D variation:

| D | Q_reg live halfs | VKQ live halfs | est. regs | minBlocksPerSM | Turing blocks/SM achievable |
|---|---|---|---|---|---|
| 64  | 16 per ncols | 16 per ncols | ~95  | **8** | 65536/(128·95)=5.4 → cap 4 |
| 128 | 32 per ncols | 32 per ncols | ~127 | **4** (3A1) | verified |
| 256 | 64 per ncols | 64 per ncols | ~180 | **2** | 65536/(128·180)=2.8 → 2 |
| 512 | 128 per ncols | 128 per ncols | ~250 | **1** (keep legacy behavior) | 1 |

Per-D launch bounds via template specialization:

```cpp
template<int D> struct vtq2_launch_bounds;
template<> struct vtq2_launch_bounds<64>  { static constexpr int blocks = 8; };
template<> struct vtq2_launch_bounds<128> { static constexpr int blocks = 4; };
template<> struct vtq2_launch_bounds<256> { static constexpr int blocks = 2; };
template<> struct vtq2_launch_bounds<512> { static constexpr int blocks = 1; };
```

**Measurement:** Nach build `cuobjdump --dump-resource-usage libggml-cuda.so | grep flash_attn_ext_vec_vtq2_cached` für jede Combo. Fail build if regs/thread > Tabelle+10%.

**ncols=2 expected reg delta:** roughly +30% auf Q_reg/VKQ/KQ_reg arrays. At D=128 ncols=2 expect ~165 regs → `minBlocksPerSM=2` fallback.

### Gate decisions

- **10 ≤ TG < 25 tok/s** → DO NOT proceed. Profile first (ncu). Likely: shmem bank conflicts, LUT residency, Q_reg spill.
