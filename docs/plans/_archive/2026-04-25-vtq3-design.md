# VTQ_3 Outlier-Channel-Split — Design + Implementation Tracker

Stand: 2026-04-25. Phase 3 der post-Gemma4 Roadmap.

## Motivation

Gemma4-26B-A4B (full-attn D=512 layers) zeigt mit VTQ_2 Trellis V-cache einen residualen `-2.4 bis -6.6% TG` slowdown vs f16, und `+0.3% PPL` Gap. Hypothese: long-tail V-Distribution Outliers werden vom 16-Codebook nicht repräsentiert, was den Lloyd-Max scale d hochtreibt und die Quantization noise auf den Bulk-Samples vergrößert.

Lösung: pro Block die K_OUT=4 grössten-Magnitude Samples raussplitten und als fp16 ground-truth speichern. Trellis quantisiert nur den verbleibenden Bulk → kleinere d → bessere Bulk-Auflösung.

## Block Layout

```c
#define VTQ_OUTLIER_K 4

typedef struct {
    ggml_half d;
    uint16_t  start_state;
    uint8_t   qs[QK_VTQ_TRELLIS * K / 8];   // K = 2,3,4 für vtq{2,3,4}_3
    uint8_t   outlier_pos[VTQ_OUTLIER_K];   // 4 B
    ggml_half outlier_val[VTQ_OUTLIER_K];   // 8 B
} block_vtq{2,3,4}_3;
```

| Type    | Block size | bpw  | vs vtq{K}_2 | Use case |
|---------|-----------:|-----:|------------:|----------|
| vtq2_3  | 48 B       | 3.00 | +0.75       | low-bpw + tail protection |
| vtq3_3  | 64 B       | 4.00 | +0.75       | balanced quality, near f16 |
| vtq4_3  | 80 B       | 5.00 | +0.75       | top-tier V quality |

## Outlier-Picking Algorithm

Pro 128-sample Trellis-Block:
1. Compute |x_i| for i=0..127.
2. argpartition → top-4 positions by magnitude.
3. Sort positions ascending (für seq-zugriff im Decoder).
4. Replace x_pos in the input vector with 0 (or with the centroid mean).
5. Run normal Trellis Viterbi on the masked vector.
6. Pack: (start_state, qs[]) + outlier_pos[] + outlier_val[].

Decoder:
1. Trellis-decode 128 samples normally.
2. For k in 0..3: out[outlier_pos[k]] = outlier_val[k].

Cost: +O(128) per encode (sort+select), +O(4) per decode (loop unroll-able).

## CUDA Decoder Notes

In FA vec kernel, after the existing per-warp `dequant_vtq2_block_4group()` produces 128 float values into `Vbuf[]`, branch on V-type and inject outliers from struct head:

```cuda
if constexpr (V_TYPE == VTQ2_3 || V_TYPE == VTQ3_3 || V_TYPE == VTQ4_3) {
    const uint8_t * pos = block.outlier_pos;
    const half2  * val = (const half2 *) block.outlier_val;  // 4 fp16 = 2 half2
    const float v0 = __low2float(val[0]),  v1 = __high2float(val[0]);
    const float v2 = __low2float(val[1]),  v3 = __high2float(val[1]);
    if (lane_id == 0) {  // single-thread overwrite, ~5 cycles
        Vbuf[pos[0]] = v0; Vbuf[pos[1]] = v1;
        Vbuf[pos[2]] = v2; Vbuf[pos[3]] = v3;
    }
    __syncwarp();
}
```

Performance impact: 4 scalar writes per block + a syncwarp. Negligible (<1% TG cost), pays for itself via better PPL.

## Risks

- Outlier-picking on CUDA encoder side requires partial-sort: not trivial but doable with warp-cooperative argmax-of-4. Falls back to deferred-V CPU path if too slow.
- 12×3×4 = 144 new template instantiations — TU build time may explode. Mitigation: only instantiate the K-V combinations used in production (3-4 K-types × 3 V-types × 2 D-values = 24).
- The VTQ_OVERLAY_FLAG path (Trick 4) is already similar and works → reuse the position-injection scaffold.
