# llama-tq roadmap — post Gemma4 unblock

Stand: 2026-04-25. Gemma4 läuft (war test-harness artifact, nicht bug). Beide Modelle (Qwen3.6-35B-A3B, Gemma4-26B-A4B) gebenchmarkt mit voller 21-config K×V matrix.

Zentrale Beobachtung: VTQ_2 Pareto auf Qwen near-lossless (-1.0% PP), aber Gemma4 hat -2.4 bis -6.6% TG slowdown. Hauptbottleneck: V_rows-loop ineffizient bei D=512 (Gemma4 full-attn) → block-header reload waste.

## Phase 3 — VTQ_3 family (Outlier-Channel-Split, "_3" Generation) 🟡 IN PROGRESS

Neuer block type. Naming-Konvention bestätigt vom user.

Status 2026-04-25: Block layouts + type enums + CLI parser + bpw resolver done. CPU encoder/decoder + CUDA dispatch pending. Smoke-Test zeigt -28-32% rel-MSE bei +0.75 bpw. Tracker: `docs/plans/2026-04-25-vtq3-design.md`.

Block layout:
```c
typedef struct {
    half       d;
    uint16_t   start_state;
    uint8_t    outlier_idx[K_OUT];                 // K_OUT=4 outlier channels
    half       outlier_vals[K_OUT * SAMPLES_PER];  // FP16 für outlier samples
    uint8_t    qs[QK_VTQ_TRELLIS * K / 8];          // K=2/3/4 für vtq2_3/vtq3_3/vtq4_3
} block_vtq{2,3,4}_3;
```

Bpw rechnung: vtq2_2 (2.25) + ~0.3 overhead = ~2.55 bpw für vtq2_3.

Hypothese: schließt den Gemma4 -6.6% TG gap auf Qwen-level (~-3%) und schließt den PPL gap zu f16 auf <0.5% bei beiden Modellen.

Aufwand-Schätzung:
- Block layout + struct definitions in ggml-common.h: 1h
- Round-trip CPU encoder/decoder + tests: 4h
- CUDA decoder + FA dispatch entries (12 K-types × 3 V-types × 4 D-werte): 1 Tag (template instantiation, build time ~30min/iter)
- Python encoder (calibration + outlier-detection on activations): 4-6h
- README + benchmarks.csv + chart: 2h

Total: 3-5 Tage für eine PR. Hoher Wert.

## Phase 4 — V-RMS-norm-aware VTQ_2 / VTQ_3 calibration

- Pure Python prep, kein kernel-touch

Aufwand: ~1 Tag. Ergänzt Phase 3 (Outlier nimmt die head-on issues, calib polished den rest).

## Phase 3.5 — VTQ_3 1-bit variant (`vtq1_3`) 📋 PROPOSED

Aktuell hat die VTQ_3-Familie nur K=2/3/4. Eine 1-bit-Variante würde Pareto-Lücke füllen:

| Type | bpw | Gap-Filler |
|---|---:|---|
| `vtq1_1` | 1.0625 | extreme low-bpw, opfert quality (PPL +16%) |
| `vtq1_3` | **1.81** | **NEU — 1-bit + 4 outliers, schließt VTQ_1-tail-collapse** |
| `vtq2_3` | 3.00 | low-bpw + tail protection |

Hypothese: `vtq1_1` D=512 leidet vor allem am long-tail (1-bit-codebook kann outliers nicht repräsentieren). 4 fp16-outlier-Slots/Block sollten den größten Anteil des PPL-gaps schließen bei nur +0.75 bpw.

**Wichtig:** VTQ_1 nutzt PolarQuant (`set_rows_cuda_pq`), nicht den Trellis-Encoder. Outlier-Pick muss in den PQ-Pfad rein, nicht Viterbi. Andere Code-Stelle (`set-rows.cu` PQ-Branch), gleiches Prinzip wie VTQ_3.

**Aufwand:** ~1 Tag. Block layout + PQ-encoder outlier-extension + CUDA decoder + FA dispatch entry + roundtrip-test.

**Blocker:** zuerst K-Collision-Bug in VTQ_2/_3 fixen (siehe `2026-04-25-vtq3-ppl-anomalies.md`) — sonst bauen wir auf bug-foundation.

## Phase 7 — imatrix-aware KTQ+VTQ Calibration (NEW, replaces Phase 4) 📋 PROPOSED

Recherche-Doc: `docs/plans/2026-04-25-ktq3-research.md`. KTQ_3 Outlier-Split wurde evaluiert und **verworfen** (+3 bpw bei QK_KTQ=32 → Pareto-dominated by ktq{b+1}_1). Trellis-K, VQ-K, per-channel-offset ebenfalls verworfen.

**Empfehlung:** model-specific Lloyd-Max codebook re-fit via mainline-imatrix tooling.

Aktuell: TQ centroids hardcoded (Laplace/Gauss). Mainline `tools/imatrix/` sammelt activation-stats — `ggml-quants.c:2353` bestätigt: "quant_weights (imatrix) is currently unused". Per-tensor calibration freischalten:

1. Extend `tools/imatrix/imatrix.cpp` to collect K/V activation variance + percentile statistics
2. New `tools/calibrate_tq_codebook.py` — Lloyd-Max re-fit per tensor → write codebook override into GGUF metadata
3. CUDA loader (`convert.cu` + `fattn-vec.cuh`) — `__constant__` table per tensor falls custom codebook present, fallback auf hardcoded
4. Subsumiert Phase 4 (Gemma4 V-pre-RMS-norm) als Spezialfall — eine Codepath für beide
5. Bench: Qwen3.6 + Gemma4, gate −0.1% PPL bei <1% TG cost

**Aufwand:** 2-3 Tage. Erwarteter Gewinn: +0.05 bis +0.2% PPL ohne bpw/speed cost.
**Risiko:** low-mid. Default-fallback auf hardcoded codebook → keine Regression.

## Phase 8 — KTQ_2 Trellis-Backbone für K-cache 📋 PROPOSED

Aktuell hat KTQ nur "_1" Generation (`KTQ1_1` … `KTQ4_1`) — alle PolarQuant + Lloyd-Max-Codebook. Eine Trellis-Variante (analog VTQ_2) würde sequentielle Korrelationen zwischen K-Samples nutzen → +1-bit-effective-precision bei gleicher bpw.

**Knackpunkt:** `QK_KTQ=32` ist zu klein für Trellis. Bei N=32:
- VTQ_2 K=2 layout: header 4B + qs 32B = 36B / 32 samples = **9 bpw** statt 2.25
- Pareto-tot — `start_state`-Header dominiert die bpw

**Lösungspfad:** `QK_KTQ` von 32 auf 128 erhöhen (alignment mit QK_VTQ). Dann passt das Trellis-Layout. Aber: gesamter K-cache-Pfad muss umgebaut werden — Block-Größen-Abhängigkeiten in `fattn-mma`, `mmvq`, `set-rows`, `convert.cu`. Großer Refactor.

**Block layout** (analog block_vtq2_2 mit QK_KTQ_TRELLIS=128):
```c
typedef struct {
    half       d;
    uint16_t   start_state;
    uint8_t    qs[QK_KTQ_TRELLIS * K / 8];   // K=2/3/4 für ktq2_2/ktq3_2/ktq4_2
} block_ktq{2,3,4}_2;
```
bpw: 2.25 / 3.25 / 4.25 (vs ktq2_1=3.5, ktq3_1=4.5, ktq4_1=5.5) → **−1.25 bpw** bei gleicher Codebook-Auflösung.

**Reihenfolge:**
1. Erst Phase 7 (imatrix-calibration) abwarten — wenn das die KTQ_1-Familie auf <0.1% PPL bringt, ist KTQ_2 unnötig.
2. KTQ_2 nur sinnvoll wenn ein Modell auftaucht wo selbst kalibrierter ktq4_1 (5.5 bpw) bei kritischem PPL-Gap landet UND Trellis-Korrelation messbaren Gewinn zeigt (PoC mit synthetischen K-stats vorab).

**Aufwand:** 1-2 Wochen (vs 1 Tag VTQ_2, weil QK_KTQ-refactor cross-cutting ist). Risiko: hoch — viele code-paths.

**Status:** deferred bis Phase 7 abgeschlossen ist und ein konkreter Bedarf gemessen wurde.
