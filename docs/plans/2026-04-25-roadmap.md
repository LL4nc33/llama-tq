# llama-tq roadmap — post Gemma4 unblock

Stand: 2026-04-25. Gemma4 läuft (war test-harness artifact, nicht bug). Beide Modelle (Qwen3.6-35B-A3B, Gemma4-26B-A4B) gebenchmarkt mit voller 21-config K×V matrix.

Zentrale Beobachtung: VTQ_2 Pareto auf Qwen near-lossless (-1.0% PP), aber Gemma4 hat -2.4 bis -6.6% TG slowdown. Hauptbottleneck: V_rows-loop ineffizient bei D=512 (Gemma4 full-attn) → block-header reload waste.

## Phase 1 — V_rows=8 D≥256 fix (in flight)

Commits 283eee6f2 + 584378082 auf turboquant. Build läuft auf gpu00.

Erwartung: Gemma4 f16/vtq2_2 von -2.4% TG → ~-1% TG. Qwen unchanged (D=128 path nicht angefasst).

Verifikation:
1. Re-bench Qwen3.6 21 configs → muss ±0.5% von vorigen Zahlen bleiben (regression-gate)
2. Re-bench Gemma4 21 configs → erwarte VTQ_1 + VTQ_2 family TG verbessert
3. README sweep tables update mit before/after delta column
4. Commit, push, doku

Falls Regression auf Qwen: revert, V_rows in templated form (D-aware) statt flat constexpr.

## Phase 2 — vtq1_1 (1bit) in den sweep aufnehmen

Mit dem V_rows fix könnte 1bit V auf D=512 plötzlich praktikabel sein. Plan:

- Same für Gemma4
- Document quality-tier-grid: "Lossless" (vtq2_2 family), "Light" (vtq2_1), "Aggressive" (vtq1_1)

## Phase 3 — VTQ_3 family (Outlier-Channel-Split, "_3" Generation)

Neuer block type. Naming-Konvention bestätigt vom user.

Block layout:
```c
typedef struct {
    half       d;
    uint16_t   start_state;
    uint8_t    outlier_idx[K_OUT];                 // K_OUT=4 outlier channels
    half       outlier_vals[K_OUT * SAMPLES_PER];  // FP16 für outlier samples
    uint8_t    qs[QK_VTQ_TRELLIS * K / 8];          // K=2/3/4 für vtq2_3/vtq3_3/vtq4_3
} block_vtq{2,3,4}_3;
```

Bpw rechnung: vtq2_2 (2.25) + ~0.3 overhead = ~2.55 bpw für vtq2_3.

Hypothese: schließt den Gemma4 -6.6% TG gap auf Qwen-level (~-3%) und schließt den PPL gap zu f16 auf <0.5% bei beiden Modellen.

Aufwand-Schätzung:
- Block layout + struct definitions in ggml-common.h: 1h
- Round-trip CPU encoder/decoder + tests: 4h
- CUDA decoder + FA dispatch entries (12 K-types × 3 V-types × 4 D-werte): 1 Tag (template instantiation, build time ~30min/iter)
- Python encoder (calibration + outlier-detection on activations): 4-6h
- README + benchmarks.csv + chart: 2h

Total: 3-5 Tage für eine PR. Hoher Wert.

## Phase 4 — V-RMS-norm-aware VTQ_2 / VTQ_3 calibration

Gemma4 normt V vor KV-write (`gemma4-iswa.cpp:92` `ggml_rms_norm`). Trellis-Codebook ist auf Qwen-V trainiert (nicht-normalisiert). Re-fit der Lloyd-Max centroids auf Gemma4 V-stats sollte den residualen PPL gap schließen.

- Pure Python prep, kein kernel-touch
- Add `tools/calibrate_vtq2.py` (existiert evtl. schon, prüfen)
- Run auf 1k V-samples aus Gemma4 forward
- Generate gemma4-specific codebook table → optional include in build oder via env var
- Bench delta

Aufwand: ~1 Tag. Ergänzt Phase 3 (Outlier nimmt die head-on issues, calib polished den rest).

## Phase 5 — Per-head adaptive bpw

Trick 2 PR1 (profiling hook) ist da (Task #137 completed). Trick 2 PR2 (per-layer mixed precision) ebenfalls (#138). Aber die per-head Logik ist nicht produktivisiert.

Plan:
- Profile Gemma4 + Qwen3.6 head-variance (already supported via `--tq-profile-heads`)
- Persist profile als gguf metadata
- Add `--tq-v-strategy adaptive` der profile.var auf bpw mapped
- High-variance heads → vtq3_3, low-variance → vtq1_1
- Bench

Aufwand: ~1-2 Tage. Marginaler Gewinn nach Phase 3+4 (vermutlich +0.2 bis +0.5% TG, -0.1% PPL) aber relativ billig.
