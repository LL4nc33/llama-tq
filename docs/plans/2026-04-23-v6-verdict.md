# v6 Verdict — Klare Strategie

**Datum:** 2026-04-23
**Input:** Alle Python-PoCs + Real-Data Roundtrip Bench

## Zentrale Messung

Auf 131k real post-RHT V-samples aus Qwen3.5-27B (QK=32):

| bits | codebook | MSE | rel MSE | bpw | Improvement vs b=2 |
|---|---|---|---|---|---|
| 2 | VTQ2_1 | 0.1104 | **11.1%** | 2.50 | baseline |
| 3 | VTQ3_1 | 0.0305 | **3.07%** | 3.50 | **3.6× besser** |
| 4 | VTQ4_1 | 0.0083 | **0.84%** | 4.50 | **13× besser** |

**Das ist der Beweis:** mehr bits für V helfen massiv. Der Sprung 2→3 alleine reduziert MSE um 72%.

## Re-evaluation der vorigen PoC

**Synthetic outlier PoC** war NICHT falsch — nur die **Interpretation**:
- "RHT entfernt outliers" ist **bezüglich Varianz** korrekt (ratio 1.50x, separation 1.01)
- **ABER**: auch wenn alle channels gleiche Varianz haben, helfen **mehr bits** auf 25% der channels trotzdem, weil MSE **nichtlinear** in bits skaliert (2² → 2³ = 4× weniger MSE-Raum pro channel)
- Ergo: Outlier-selection muss **nicht statistisch** begründet sein. Selbst **random 25% channels** @ 3-bit würde den Durchschnittsgewinn liefern

## Überarbeitete Strategie

### Option A: Mixed-Precision V-Cache (NEUE Priorität #1)

Implementiere **VTQ2_1_MIXED**: 32 channels @ 3-bit + 96 channels @ 2-bit (eff 2.25 bpw).
Erwartete rel MSE: Interpolation zwischen 11% und 3% → **~9%** (18% Verbesserung vs uniform VTQ2_1).

**Block-Layout:**
```c
typedef struct {
    ggml_half d;               // block scale
    uint8_t qs_mix[40];        // packed: first 32 samples @ 3-bit (12 bytes),
                               // remaining 96 samples @ 2-bit (24 bytes) = 36 bytes
                               // + 4 bytes padding for alignment = 40 bytes
} block_vtq_mixed;
```

Actual: bei QK=128 head_dim, 32 outlier × 3 bit + 96 regular × 2 bit = 96 + 192 bits = **288 bits = 36 bytes**. Plus fp16 scale = **38 bytes** für 128 samples = **2.375 bpw**.

Vergleich VTQ2_1 (QK=32, 2 bytes scale + 8 bytes qs = 10 bytes / 32 samples = **2.5 bpw**).

**Ergo: VTQ_MIXED ist SOWOHL bpw-cheaper ALS AUCH accuracy-besser als VTQ2_1.** Das ist ein Free Lunch im Paper-Sinn.

### Option B: Simple Upgrade zu VTQ3_1 als Production Default

VTQ3_1 ist **schon implementiert** in unserer codebase. Migration:
- Kernel ready
- CUDA path ready
- Roundtrip unit tests ready

**Trade-off:** 4.5 bpw statt 2.5 bpw (80% mehr V-cache memory). Für 400k ctx heißt das: von 12 GB → 22 GB VRAM (passt nicht).

**Ergo Option B nur nützlich wenn:**
- Kontext < 200k (dann passt es)
- User legt Wert auf Accuracy über Kontext-Länge

## Priorisierung

1. **VTQ_MIXED** (neu): High ROI, ähnliche bpw, deutlich bessere Accuracy. 3-5 Tage effort.
2. **VTQ3_1 als Option**: Zero-code-change, nur docs + production config + --cache-type-v flag. 1 Tag.
3. **C1 Streaming-Window**: Decode-Latency benefit, unabhängig von bpw. 2-3 Tage.

## Offene Fragen

- **Welche 32 channels?** Synthetic test zeigt random ≈ calibrated (weil RHT macht uniform). Also zufällige Selektion spart Calibration-Complexity.
- **Welche Pack-Order?** Alternierend (out,reg,out,reg,...) oder getrennt (alle out zuerst dann alle reg)?
- **Per-head oder global outlier-set?** Global ist einfacher für Kernel-Dispatch, per-head könnte besser für GQA-heads.
