User directive. 35B runs take 5-8h per config — way too slow for

Small model, ~6s for f16 baseline on 5 chunks. Use it as the

User clarified the real goal: **IQ2-sized weights with Q8 quality**.
This is what the TurboQuant paper (arXiv:2504.19874) actually
describes — we've been implementing the KV-cache side only.

### CUDA encoder opts B+C — one win, one revert

**Commit `5939f05ed` (fused dp_next re-init + writeback)**: kept.
Eliminates redundant global R/W every step, correctness verified.

**Commit `a1a4eb425` (__ldg on LUT)**: reverted in `9d526db23`.
Turing RO cache is 48 KiB/SM, LUT is 256 KiB. `__ldg` forced
~20% hit rate with L2 refill stalls on atomicMin critical path.
Old plain load went straight to L2 (~100% hit after warmup).
Observed regression: VTQ3_2 on 0.8B went from 135s to 300s+.

Lesson for future: `__ldg` only helps when working set fits in
RO cache (48 KiB Turing, 128 KiB Ampere). For random-access
tables > RO cache size, L2 locality beats RO cache pollution.

### Codebook research: TQ3 > TQ2

Second researcher warned: at 2 bpw, RHT+1D-Lloyd is theoretical
wash vs IQ2_XXS (IQ2's 8-D E8 packing gain ≈ TurboQuant's
gaussianization gain). Real win emerges at 3+ bpw where 1-D Lloyd
approaches its asymptote.

Recommendation: make **TQW3** (3 bpw) the flagship weight quant,
not TQW2. Position vs IQ4_XS (4.25 bpw): similar PPL with 25%
smaller footprint.

Agent investigation found that **KTQ3_1 (4.5 bpw)** already has

Path chosen: add `LLAMA_FTYPE_MOSTLY_TQW3` + routing in
