**Commit `5939f05ed` (fused dp_next re-init + writeback)**: kept.

**Commit `a1a4eb425` (__ldg on LUT)**: reverted in `9d526db23`.

### Codebook research: TQ3 > TQ2

Second researcher warned: at 2 bpw, RHT+1D-Lloyd is theoretical
wash vs IQ2_XXS (IQ2's 8-D E8 packing gain ≈ TurboQuant's
gaussianization gain). Real win emerges at 3+ bpw where 1-D Lloyd
approaches its asymptote.

Recommendation: make **TQW3** (3 bpw) the flagship weight quant,
not TQW2. Position vs IQ4_XS (4.25 bpw): similar PPL with 25%
smaller footprint.

## TQW3 Sprint Result: BROKEN — debugging needed

Agent landed 4 commits (dd2c01a50, 349e0272c, dba2361d9, 5f8dd3e30)
implementing TQW3 as KTQ3_1 ftype alias. Pipeline builds, quantize
produces a GGUF file (0.8B → 501 MB, 35% smaller than q8_0's 775 MB).

**PPL on TQW3-quantized 0.8B: 2.3 billion.** Completely broken output.

### Root cause hypothesis (from agent's own analysis, commit 5f8dd3e30)

> FA path on K-cache uses old convention and works end-to-end,
> suggesting self-consistent 'push to Q' trick that does NOT apply
> to row dequant for weights.

KTQ3_1 is designed for K-cache quantization where the FA kernel
applies FWHT to **Q (query)** instead of inverse-FWHT to K — the
"v7 Hadamard-domain K·Q dot" trick at `turboquant.cuh:55-75`. This
makes K·Q work correctly while leaving K physically stored in
Hadamard domain.

For **weights in mul_mat**, activations are in physical domain, so
the dequant MUST apply the full inverse RHT (inverse FWHT + sign
flip). Agent's sign-flip commit (5f8dd3e30) only addressed a symptom,
not the missing inverse FWHT.

### Decision: Option A (ftype alias) is structurally wrong

The whole premise of "reuse KTQ3_1 format as-is" doesn't work —
KTQ3_1 block data is stored in Hadamard domain, which mul_mat can't
consume without an inverse-transform dequant kernel that doesn't
exist.

Real fix options:
1. **Add a row-dequant-with-full-inverse-RHT kernel for KTQ3_1**
   (~200 LOC new CUDA). Simple wrapper around existing machinery.
2. **New type TQW3 with weights stored in physical domain** — quantize
   applies RHT → Lloyd → dequant applies Lloyd-lookup → inverse-RHT
   → write physical values. Standard mul_mat works unchanged.

Option 2 is cleaner and is what a real TurboQuant weight-quant should
look like per the paper.
